# STYLE_GUIDE.md

> **Єдиний стандарт оформлення документації Artificial Life Engine**

---

# Призначення

Цей документ визначає правила написання всієї документації проєкту.

Мета документа:

* зробити документацію послідовною;
* спростити навігацію;
* зробити її зрозумілою як людям, так і AI-агентам;
* мінімізувати неоднозначності;
* забезпечити масштабованість проєкту.

Ці правила застосовуються до всіх документів каталогу `docs/`.

---

# Основні принципи

## Документація важливіша за код

Документація описує модель світу.

Код лише реалізує її.

Якщо реалізація суперечить документації — необхідно:

1. або виправити код;
2. або змінити документацію через ADR.

---

## Один документ — одна тема

Кожен файл повинен описувати лише одну логічно завершену тему.

Приклади:

✔️

```
cell.md
```

```
resources.md
```

```
energy.md
```

❌

```
cell-and-energy.md
```

---

## Документи повинні бути незалежними

Документ повинен пояснювати тему повністю.

При цьому дозволяється посилатися на інші документи замість дублювання інформації.

Наприклад

```
Детальний опис Tick знаходиться у world/tick.md
```

а не повторювати весь документ.

---

# Структура документів

Усі Canon-документи повинні мати приблизно однакову структуру.

```text
# Назва

## Призначення

## Основні поняття

## Правила

## Обмеження

## Приклади

## Пов'язані документи

## ADR

## TODO / Open Questions
```

Не всі розділи є обов'язковими.

Якщо розділ не потрібний — його можна пропустити.

---

# Типи документів

## Canon

Описує правила світу.

Canon містить лише:

* прийняті рішення;
* стабільні визначення;
* офіційну специфікацію.

Canon не містить:

* гіпотез;
* статей;
* сумнівних рішень.

---

## Research

Research використовується для:

* експериментів;
* альтернатив;
* порівнянь;
* результатів досліджень;
* ідей.

Research не є специфікацією.

Будь-яке рішення з Research повинно пройти через ADR перед перенесенням у Canon.

---

## ADR

ADR пояснює історію рішення.

Не описує алгоритм.

Не описує API.

Не дублює Canon.

---

# Правила написання

## Писати українською мовою

Документація ведеться українською.

Код залишається англійською.

---

## Назви класів

У документації:

```
Cell
Joint
Genome
Material
Resource
Tick
```

можна використовувати англійські назви, якщо саме вони використовуються у коді.

Пояснення при цьому пишуться українською.

---

## Не використовувати неоднозначні слова

Погано

```
речовина
```

Добре

```
Resource
```

або

```
Material
```

---

## Один термін — одне значення

Наприклад

```
Resource
```

ніколи не означає Material.

Якщо виникає нове поняття — його необхідно додати до `GLOSSARY.md`.

---

# Формат правил

Правила повинні бути короткими та перевіряними.

Добре

```
Енергія не транспортується між клітинами.
```

Погано

```
Бажано не переносити енергію.
```

---

## Уникати припущень

Писати лише те, що є правилом.

Не використовувати слова

```
можливо
напевно
ймовірно
```

у Canon.

Для цього існує Research.

---

# Приклади

Canon-документи повинні бути короткими.

Короткий приклад на 3-5 рядків дозволений, якщо він уточнює правило на місці.

Довгі сценарії, traces, config examples і walkthrough треба переносити в `docs/examples/`.

Приклад не є правилом. Якщо приклад суперечить Canon-документу, правильним вважається Canon.

Canon не повинен містити навчальні історії, повтори або демонстраційні сценарії, які можна винести в examples.

---

# Відкриті питання

Якщо питання ще не вирішене, воно оформлюється окремим блоком.

```
## Open Questions

- ...
```

або

```
## TODO

- ...
```

Не допускається змішування відкритих питань із затвердженими правилами.

---

# Посилання

Наприкінці документа рекомендується вказувати:

```
## Пов'язані документи

- world/resources.md
- world/materials.md
- biology/cell.md
```

---

# Правила для AI-агентів

Перед внесенням будь-яких змін агент повинен:

1. прочитати `PRINCIPLES.md`;
2. перевірити `GLOSSARY.md`;
3. знайти відповідний Canon-документ;
4. перевірити ADR;
5. лише після цього пропонувати зміни.

Якщо потрібне правило відсутнє:

* не вигадувати його;
* додати TODO;
* або створити новий ADR.

---

# Правила внесення змін

## Невеликі уточнення

Можна змінювати без ADR.

Наприклад:

* виправлення формулювання;
* друкарські помилки;
* уточнення прикладів.

---

## Фундаментальні зміни

Обов'язково створюється ADR.

Приклади:

* новий закон світу;
* зміна моделі геному;
* зміна моделі енергії;
* новий тип взаємодії;
* зміна фізики.

---

# Позначення статусів

Використовуються такі статуси:

| Статус        | Значення                              |
| ------------- | ------------------------------------- |
| Stable        | Повністю прийняте рішення             |
| Mostly Stable | Майже завершене, можливі дрібні зміни |
| Draft         | Документ у процесі написання          |
| Research      | Відкрите дослідження                  |
| Deprecated    | Застаріле рішення                     |
| Rejected      | Відхилена ідея                        |

---

# Стиль оформлення

Рекомендується:

* короткі абзаци;
* списки замість довгого тексту;
* схеми ASCII для пояснення структури;
* мінімум декоративного оформлення;
* однакова структура документів.

---

# Принцип еволюції документації

Документація створюється зверху вниз.

```
README
    ↓
PRINCIPLES
    ↓
World
    ↓
Biology
    ↓
Engine
```

Фундаментальні документи повинні змінюватися якомога рідше.

Чим нижче рівень документації, тим більше допускається деталізація та уточнення.

---

# Головне правило
